## 1. What This API Does

This API provides programmatic access to the VA's Computerized Patient Record System (CPRS) data by communicating with a VistA server via the XWB broker protocol. It mirrors every capability of the desktop CPRSChart.exe application.

**Key capabilities:**
- Patient search, selection, and demographics
- Clinical notes (TIU documents) — create, read, sign, addend
- Orders — medications, labs, radiology, diets, consults
- Problem list management
- Allergies and adverse reactions
- Vitals/measurements
- Lab results (cumulative, interim, micro)
- Reports (health summaries, imaging, surgery)
- Clinical reminders and health maintenance
- Discharge summaries
- Encounter/visit management (PCE)
- Notifications and alerts

## 2. Authentication Flow

All endpoints except `/api/connection/sites` require a JWT Bearer token.

### Step 1: Get Available Sites
```
GET /api/connection/sites
→ Returns: [{"site_id": "128", "name": "Mock VistA", "visn_name": "VEHU"}]
```

### Step 2: Authenticate (choose one)
```
GET /api/connect?HostName=localhost&Port=9200&SiteId=128&AccessCode=cprs&VerifyCode=cprs1234
→ Returns: JWT token string (plain text, not JSON)

GET /api/connectbysite?SiteId=128&AccessCode=cprs&VerifyCode=cprs1234
→ Returns: JWT token string (resolves host/port from site config)
```

### Step 3: Use Token
```
Authorization: Bearer <jwt_token>
```
Include this header on all subsequent requests. Token expires in 20 minutes.

### Step 4: Heartbeat (keep session alive)
```
GET /api/session/pulse
→ Returns: true
```

### Step 5: Disconnect
```
POST /api/session/disconnect
→ Returns: {"status": "disconnected"}
```

## 3. Common Workflows

### Review a Patient's Chart
1. `GET /api/connection/sites` → pick site
2. `GET /api/connectbysite?SiteId=128&AccessCode=...&VerifyCode=...` → JWT
3. `GET /api/patient/search/last5?last5=C1234` → find patient (last initial + last 4 SSN)
4. `GET /api/patient/select?dfn=<DFN>` → select patient, get demographics
5. Now call any patient-context endpoints using `dfn=<DFN>`

### Read Clinical Notes
```
GET /api/note/list?dfn=3&context=1&early=3250101&late=3260521
→ Returns: list of TiuDocument objects (structured JSON)

GET /api/note/text?ien=123
→ Returns: ["Line 1 of note text", "Line 2..."]
```

### View Active Orders
```
GET /api/order/list?dfn=3&filterTS=2&dGroup=ALL
→ Returns: list of Order objects (structured JSON)

GET /api/order/detail?dfn=3&ien=456
→ Returns: ["Detailed order text lines..."]
```

### Check Active Problems
```
GET /api/problem/list?dfn=3&status=A
→ Returns: list of Problem objects (structured JSON)
```

### Get Lab Results
```
GET /api/lab/recent?dfn=3
→ Returns: ["Recent lab result lines..."]

GET /api/lab/cumulative?dfn=3&daysBack=365
→ Returns: ["Cumulative report lines..."]
```

### View Allergies
```
GET /api/coversheet/allergies?dfn=3
→ Returns: ["Allergy list lines..."]
```

### Check Clinical Reminders
```
GET /api/reminder/applicable?dfn=3
→ Returns: ["Reminder status lines..."]
```

## 4. Response Format Conventions

### Snake_case JSON
All structured responses use **snake_case** field names:
```json
{"date_of_birth": "1945-03-15T00:00:00", "location_name": "WARD 3B"}
```

### Two Response Types

**Structured (Pydantic models)** — parsed into typed JSON objects:
- Patient demographics, search results, sensitivity checks
- User info
- Orders list
- Consults list
- Problems list
- TIU documents list (notes, DC summaries)
- Report definitions

**Raw strings** — VistA wire format passed through as `string` or `list[string]`:
- Note text, order detail, lab reports, consult detail
- Most action/mutation results (sign, save, delete)
- Lookup/subset lists

Raw strings often use VistA delimiters: `^` (caret) for fields, `~` for sub-records.

## 5. Key Parameters

| Parameter | Description | Example |
|-----------|-------------|---------|
| `dfn` | Patient DFN (internal entry number in VistA) | `"3"` |
| `ien` | Internal entry number for a record (note, order, consult) | `123` |
| `orderId` | Order IFN (internal file number) | `"456;1"` |
| `location` | Hospital location IEN | `68` |
| `provider` | Provider DUZ (user IEN) | `83` |
| `esCode` | Electronic signature code (encrypted) | `"ABC123"` |
| `early`/`late` | Date range in FM format (YYYMMDD) | `"3250101"` |
| `alpha`/`omega` | Alternate date range params | `"3250101"` |
| `context` | View context (1=All, varies by endpoint) | `1` |
| `dGroup` | Display group for orders (ALL, meds, labs, etc.) | `"ALL"` |
| `startFrom`/`direction` | Paging: start text and direction (1=forward, -1=back) | `"A"`, `1` |

### FileMan Date Format
VistA uses FileMan dates: `YYYMMDD.HHMMSS` where YYY = year - 1700.
- `3260521` = May 21, 2026
- `3260521.143000` = May 21, 2026 at 2:30 PM

## 7. Clinical Domain Glossary

| Term | Description |
|------|-------------|
| **CPRS** | Computerized Patient Record System — the VA's primary EHR interface |
| **VistA** | Veterans Health Information Systems & Technology Architecture — the backend system |
| **DFN** | Data File Number — patient's unique internal ID in VistA |
| **DUZ** | User's internal ID (provider/clinician) |
| **IEN** | Internal Entry Number — generic record ID |
| **TIU** | Text Integration Utilities — the notes/documents subsystem |
| **PCE** | Patient Care Encounter — structured encounter data (diagnoses, procedures, vitals) |
| **CWAD** | Crisis/Warnings/Allergies/Directives — patient safety flags |
| **FM Date** | FileMan date format: YYYMMDD.HHMMSS (YYY = year - 1700) |
| **RPC** | Remote Procedure Call — VistA's API protocol |
| **XWB** | Broker protocol for VistA communication |
| **DC Summary** | Discharge Summary — clinical document for hospital discharge |
| **OI** | Orderable Item — an item that can be ordered (drug, test, procedure) |
| **Quick Order** | Pre-configured order template for common orders |
| **Order Check** | System validation before an order is placed (drug interactions, duplicates) |
| **e-Signature** | Electronic signature code required to sign notes and release orders |
| **SSO** | Single Sign-On token (alternative to access/verify codes) |

## 9. Error Handling

| HTTP Code | Meaning |
|-----------|---------|
| 400 | Missing required parameter (e.g., SiteId, dfn) |
| 401 | Invalid/expired JWT or authentication failure |
| 403 | Insufficient VistA permissions |
| 500 | VistA RPC error or connection failure |

Error responses are JSON: `{"detail": "error message"}`
